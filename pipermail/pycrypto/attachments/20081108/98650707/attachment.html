The first test I wrote may be a minor improvement on the simple test, as it merely ensures that the value returned by the rng is not always the same. (it is plausible that an rng might return the same data twice).<br><br>For the second test I chose a large sample so that the average of all the data would be roughly 128, I chose the large variation sort of out of the air, although the average of a large sample should (statistically speaking) be very close to 128. After running about 10 tests the values were never outside of 128 +/- 2.<br>

I&#39;m sorry if I&#39;m telling something stupid, since I was not working with RNGs analysis for about 18 years, but from what I still remember. There was a test (in fact, series of tests) that surprisingly failed lot of these old times RNGs. It&#39;s sort of N-dimensial bins histogram.<br>


<br>
The simplest was easy. Let&#39;s talk pairs of N-bit randoms as (x,y) coordinates in a finite rectangle (using N or mod) and see whether they would tend to cluster. I&#39;ve seen in my own eyes that one RNG that was supposed to be sufficient, was constantly generating a few small triangles leaving the rest clear :)<br>


<br>
So, the general test we were performed was kind of following. Let&#39;s take N-dimensional cube and fill it with RNG series of size L, then check how even it fills the cube. Simplest case is to divide it to equal clusters (bins), so there should be, say, L/1000 clusters least, calculate how many pseudo-random points hit each bin and see how plain is this histogram. It should not vary too much if the RNG is ok and L is big. We can, say, check the standard deviation. Better to check several bins size, using various random primes as factors for bins sizes. Then increment N and check it again.<br>


<br>
I&#39;ve read in late 80s that such test failed on some very famous RDBMS system of these old good times (as I remember, all dots filed 64 plans in the 3(?)D cube or so).<br>
<br>
For sure there must be a plenty of better tests by now, but I&#39;m not familiar with them.<br>
